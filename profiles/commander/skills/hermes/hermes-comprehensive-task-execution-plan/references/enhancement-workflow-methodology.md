# 全系统增强改造方法论 — 五步法（2026-06-01 由实战固化）

任何"对已有系统做大规模增强"的任务（非从零开发），必须执行此流程。

## Step 1: 全功能彻底盘点

检查当前所有模块的健康状态，用真实命令输出，不能凭记忆。

```
采集:  raw量/cron条数/24h采集量/采集源数/错误率
评分:  覆盖百分比/评分分布/低分待归档量
推送:  总计/今日/错误率
记忆:  semantic/scene/profile/episodic/structmem各多少条
上下文: 6脚本存在性/cron运行/索引可追溯/输出新鲜度
齿轮:  gear_health/heartbeat
自进化: 进化报告天数/13模块存在/复盘记录数
多Agent: 130员工/390专家存在性
增强模块: P0-P3 15个文件全部存在
安全/调优/反馈: 全部存在
模型配置: v4-flash/v4-pro/chat已配
```

## Step 2: 全网调研方法论（用delegate_task并行）

至少3个并行调研方向，每个输出：
- 方法论名称
- 核心机制（1-2句话）
- 可借鉴点（具体）
- 价值评分(1-10)

## Step 3: 制定分层优先级计划

```
P0(立即): 本轮必须完成的底层增强
P1(本周): 核心功能强化
P2(下周): 质量/审核体系
P3(后续): 高级进化能力
```

每条必须有：
- 实现路径（具体文件路径）
- 验证方法（具体命令）
- 优先级（P0-P3）
- 时间估算

## Step 4: 对抗性审查

实施前用这些问题审查计划：
- 每条改进能否完全自动运行？还是需要人手动触发？
- 是否有依赖外部条件？断了怎么办？
- 与现有系统会冲突还是补充？
- 如果Hermes今晚不工作，cron能自己跑起来吗？

## Step 5: 从底层逐层向上实施

- 先改核心架构，再改上层功能
- 每层完成后立即测试验证
- 新增文件写到~/.hermes/下的对应目录
- 改造已有系统时：保留不重构，只补充不推翻
- 每层交付时提供三级证据（实时可查 > 日志证明 > 声称通过）
