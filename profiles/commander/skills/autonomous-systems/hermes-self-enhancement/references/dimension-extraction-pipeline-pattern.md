# 维度提取管道模式 (Dimension Extraction Pipeline)

## 适用场景
从大规模AI生成文本数据（如to_chi 52万条）中，按维度拆解并重建高质量维度库。

## 数据源特征
- to_chi数据：javrate（13.5万条色情写真）+ xhs（38.5万条小红书混合）
- 每条数据格式：一段自然描述 + 摄影参数标签
- 含52万条数据，261个文件（68个javrate + 193个xhs）

## 排除规则（xhs专用）
- ❌ 室内设计：以"现代/简约/极简/北欧..."开头描述房间的
- ❌ 美甲：纯指甲特写无人物
- ❌ 食物：纯食物摄影无人物用餐场景

## 三版本迭代法

### v1: 关键词匹配（广度优先）
- 16个大类，每个大类用预定义关键词列表在原文中匹配
- 输出：纯关键词（如"自然光""白衬衫"）
- 优点：速度快（260文件/5分钟），覆盖度广
- 缺点：唯一值少（如光照只有27种）

### v2: 原文片段提取（深度优先）
- 提取包含关键词的上下文片段（±15字）
- 缺点：片段太长太杂，包含无关上下文，重复度高

### v3: 子维度细分（最佳方案）
- 将16个大类拆成26个子维度（如01_发型/01_姿势/02_上装/02_下装）
- 每个子维度用精准规则提取+原文保留
- 输出：26个子维度库，47.5万条提取
- 清洗后缩减到各库10-473条干净条目

## 清洗流程（cleaning.py）
1. 去除前缀杂质（"一位""留着""穿着"等）
2. 分类标准化（发型归一：黑长发/棕短发/卷发；姿势归一：站立/坐着/躺着）
3. 去重保留唯一值
4. 过滤短于2字或无意义条目

## 关键陷阱
1. **to_chi数据质量有限**：AI生成的批量描述文本，丰富性不足。维度库适合与100个prompt模板结合使用，不适合单独生成prompt。
2. **原文片段提取的双刃剑**：保留上下文会提高丰富度但引入杂质。需要后续清洗。
3. **排除规则必须保守**：宁可多提取一些低质量条目，也不把有效数据误排除。
4. **子维度的粒度**：太细（如30个子维度的01_发型）反而无用；太粗（如只有"发型"一个类）丢失区分度。找到20-30个细分类的平衡点。
