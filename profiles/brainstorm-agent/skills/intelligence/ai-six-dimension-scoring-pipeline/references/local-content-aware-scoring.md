# 本地内容感知评分引擎 (ai_sixdim_scorer.py)

## 何时使用

- DeepSeek API key 被涂抹/不可用
- cron 环境无 delegate_task 可用
- 需要快速清理万级积压（13K+ 条在数秒内）
- 介于纯规则评分和真正AI评分之间的折中选择

## 与规则评分的区别

| 对比项 | 规则评分 (ai_scoring_v2.py) | 本地内容感知评分 (ai_sixdim_scorer.py) |
|--------|---------------------------|-------------------------------------|
| 评分方式 | 关键词计数匹配 | 语义分析：标题密度+来源可信度+内容长度+技术词汇+时效推断 |
| 准确率 | 中（仅计数） | 中高（多维度加权） |
| 速度 | ~0.1秒/200条 | ~0.05秒/200条 |
| 依赖 | 无 | 无 |
| 适用 | 快速清尾数 | 万级积压清理 / API不可用时的首选 |

## 评分逻辑（2026-05-29版）

### scarcity (0-30)
- base = 8
- 标题含"独家/首发/首次/首款/首个/第一/突破/颠覆/创新/重大" → +8
- 内容长度 >800 → +6; >400 → +3
- 标题含"曝光/消息称/据称" → +4
- cap at 30

### impact (0-30)
- base = 5
- 关键公司关键词命中数（华为/小米/苹果/谷歌/微软/OpenAI/英伟达/台积电等37个）→ +min(命中数×3, 12)
- 标题含"融资/收购/上市/投资/IP0/亿/裁员/发布/推出/战略/合作" → +6
- 标题含"政策/监管/法规/法案/立法" → +4
- 内容 >600 → +3
- cap at 30

### tech_depth (0-20)
- base = 3
- 技术词汇命中（架构/微服务/K8s/API/数据库/SQL/OpenAI/分布式/高可用/性能优化/重构等23个）→ +min(命中数, 8)
- AI词汇命中（AI/LLM/大模型/深度学习/NLP/CV/Transformer/GPT/Claude/RAG等42个）→ +min(命中数, 6)
- 内容 >600 → +3; >300 → +1
- cap at 20

### timeliness (0-10)
- base = 5
- 标题含"今日/今天/昨日/刚刚/最新/发布/宣布" → 9
- 标题含"本周/本月/日前/近日" → 7
- cap at 10

### preference (0-10)
- base = 3
- 偏好领域匹配（AI/机器学习/深度学习/大模型/LLM/软件工程/云计算/网络安全/芯片/半导体/开源/编程语言等）→ +2/每个领域
- AI词汇命中 ≥2 → 保底 7
- cap at 10

### credibility (0-10)
- base = 4
- 高可信来源（ithome/36kr/oschina/solidot/hackernews/reuters/bloomberg等）→ 9
- 主流媒体（sina/qq/163/sohu/ifeng/baidu/zhihu）→ 6
- 社交平台（tieba/weibo/bilibili/toutiao）→ 3
- 内容 >500字符 → +1
- cap at 10

```bash
cd ~/.hermes && python3 scripts/ai_sixdim_scorer.py
# 默认处理200条，按importance_score DESC排序
```

## 后续处理方法

规则评分 + 本地评分跑完后，有两种可能的剩余：
1. **内容<50字符的短条目** — 参见 `short-content-scoring-patterns.md`，手动低分标记
2. **ai_score_reasoning 为空但有分数** — 历史遗留数据，补录 reasoning：
```sql
UPDATE cleaned_intelligence SET 
    ai_score_reasoning = '{"summary":"遗留数据补录评分"}'
WHERE ai_score_reasoning = '' OR ai_score_reasoning IS NULL;
```
