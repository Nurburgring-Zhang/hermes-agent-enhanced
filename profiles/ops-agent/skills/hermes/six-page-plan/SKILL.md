# Six-Page Plan (6页纸规划模板)

源自Amazon内部工作方法，用于在复杂任务执行前进行结构化规划。通过6个标准章节迫使思考全面深入，避免遗漏关键维度。

## 模板格式

### 第1页: 背景与问题定义
- **业务/技术背景**: 这个任务源自什么场景？为什么现在要做？
- **问题陈述**: 用一句话精确描述要解决的核心问题
- **影响范围**: 哪些系统/模块/用户会受到影响？
- **利益相关方**: 谁关心这个任务的成败？

### 第2页: 解决方法
- **方案概述**: 拟采用的核心方案（1-3句话）
- **方案选项**: 至少列出2-3种可选方案，说明选择理由
- **预期效果**: 方案落地后预期达到的效果
- **技术栈/工具**: 需要用到哪些技术栈或现有工具

### 第3页: 详细实施计划
- **阶段划分**: 将任务分解为3-7个阶段
- **每个阶段**: 输入条件、执行步骤、输出产物、验收标准
- **时间预估**: 每个阶段的预估工时
- **依赖关系**: 阶段之间的前置依赖

### 第4页: 关键指标和成功标准
- **成功定义**: 如何定义这个任务"做完了"？
- **测量指标**: 至少2个可量化的成功指标
- **验收检查清单**: 逐项检查项（可测试/可验证）
- **最低可接受标准**: 什么情况下算"及格"？

### 第5页: 风险与回滚计划
- **风险识别**: 至少列出3个潜在风险
- **风险等级**: 每个风险的概率/影响评估（高/中/低）
- **缓解措施**: 每个风险的预防/应对方案
- **回滚方案**: 如果方案失败，如何回滚到之前状态？
  - 代码回滚: git revert / 备份还原
  - 数据回滚: 数据库快照恢复
  - 配置回滚: 备份配置恢复

### 第6页: 附录
- **参考资料**: 相关文档、链接、代码位置
- **关键决策记录**: 为什么做出某些关键选择
- **未尽事宜**: 本轮不处理但需要记录的问题
- **历史复盘链接**: 指向相关复盘报告的链接

## User Preference: 先看报告再执行

格林主人偏好：**先看完整审计报告，再决定是否执行优化。** 任何"审计→优化"类任务，第一步必须输出完整状态报告（带量化数据），等人确认后再动手改。

流程:
1. 输出当前状态报告（模块健康/数据指标/优化项分级）
2. 标注 "是否要执行？"
3. 等确认后再开始编码

反模式: 直接开始修，修完了才报告。必须先报告。

## 使用方法

在复杂任务（预计>5步或跨多个系统）执行前**强制调用**此Skill：

1. **调用方式**: `skill_view(name='six-page-plan')` 读取模板
2. **输出文件**: 结构化规划文档存入 `reports/plans/<task_id>.md`
3. **引用机制**: 在执行过程中随时打开规划文档检查方向是否偏离

## 集成到任务执行流程

每次复杂任务前自动调用 — 通过gear_enforcer.py的V3循环集成：

1. **触发时机**: 当 `get_active_task()` 返回的任务预计超过5步时
2. **自动生成**: 调用此Skill → 读取模板 → 填充内容 → 写入 `reports/plans/<task_id>.md`
3. **进度对齐**: 每阶段结束时比对规划文档，检查是否偏离
4. **Checkpoint关联**: 规划文档的task_id与checkpoint_recorder.py共享同一task_id
