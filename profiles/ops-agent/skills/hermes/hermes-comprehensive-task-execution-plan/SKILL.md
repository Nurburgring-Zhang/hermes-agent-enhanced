### 0.4 穿透验证铁律（2026-06-01 由实战固化）

格林主人对"已实现"声明的验收标准极端严格。**声明"完成"时，必须穿透验证到对话层生效级别，不能只到文件存在或cron在跑。这是本轮对话的核心教训：**

**穿透验证的四个层级（缺一不可）：**
```
层级1: 文件存在          → stat/文件大小/未损坏
层级2: 自动运行(cron)    → crontab -l | grep <脚本名>
层级3: 输出数据正确     → 执行脚本验证输出非空/内容正确
层级4: 对话层生效        → 确认系统真的在用这个输出，而不是文件写在那没人读
```

**本轮实战案例——声称"完全实现"但穿透验证发现的问题：**
| 声称 | 层级1(文件) | 层级2(cron) | 层级3(数据正确) | 层级4(生效) |
|------|:---:|:----:|:--------:|:------:|
| surgical_context_slicer 按任务切分 | ❌ 只剩字节码 | ❌ 无cron | ❌ | ❌ |
| context_auto_assoc 自动关联 | ❌ 只剩字节码 | ❌ 无cron | ❌ | ❌ |
| context_index 可追溯 | ✅ 文件存在 | ❌ cron覆盖后sections=0 | ⚠️ sections=0 | ❌ |
| 5个上下文cron | — | ❌ 全不在crontab | — | ❌ |

**验证命令模板（每次交付时使用）：**
```
# 文件完整
stat -c%s ~/.hermes/scripts/<name>.py
# cron运行
crontab -l | grep <name>
# 输出新鲜度
stat ~/.hermes/reports/<output>.json
# 内容验证
python3 -c "import json; d=json.load(open('...')); assert len(d['sections'])>0"
# 对话层生效——当前对话是否真的在用索引版SOUL.md而不是全量
head -5 ~/.hermes/SOUL.md
```

### 0.5 增强/改造类任务的五步法

任何"增强一个已有系统"的任务（非从零开发），必须执行此流程：

```
Step 1: 全功能彻底盘点
  - 检查当前所有模块的健康状态（cron/数据库/脚本/齿轮/记忆/Agent）
  - 逐个模块列出现状指标（数据量/连接状态/最后运行时间/错误率）
  
Step 2: 全网调研方法论
  - 至少3个并行调研方向并行用delegate_task执行
  - 每个方向：找核心论文/开源项目/成功案例，提炼核心机制
  - 调研产出：每个方法论→核心机制→可借鉴点→价值评估(1-10)
  - 必须有真实来源引用，不能靠"我知道"

Step 3: 制定分层优先级计划
  - P0(立即): 本轮对话必须完成的底层增强
  - P1(本周): 核心功能强化
  - P2(下周): 质量/审核体系
  - P3(后续): 高级进化能力
  - 每条必须有：①实现路径(具体文件路径) ②验证方法 ③优先级 ④时间估算

Step 4: 对抗性审查（格林主人"能不能完全自动化"测试）
  在实施前用这些问题审查计划：
  - 每条改进能否完全自动运行？还是需要人手动触发？
  - 是否有依赖外部条件（API/网络/GPU）？断了怎么办？
  - 与现有系统会冲突吗？还是补充？
  - 如果Hermes今晚不工作，cron能自己跑起来吗？

Step 5: 从底层逐层向上实施
  - 先改核心架构，再改上层功能
  - 每层完成后立即测试验证（不等到全部做完）
  - 新增文件写到~/.hermes/下的对应目录
  - 改造已有系统时：保留不重构，只补充不推翻
```

**🔴 信息误差累积陷阱（2026-06-01 核心教训）**

格林主人明确指出：**每轮都有信息误差，1000轮会累积到可怕的程度。** 这不是段切换的问题，是每轮对话本身的信息损失。
解决方案不是"让我记得更准"，而是让记忆变成可选项——所有关键信息都在文件系统里，我只读不记。

```
误差类型A: "我记得"误差 — AI从上下文回忆, 可能记错
  解决: 每步执行结果写入文件系统层面, 不是记忆层面
        terminal命令的执行结果 → tool_unloader写到refs/*.md
        齿轮状态 → 每分写入wake_guide.json
        段状态 → 每轮写入segment_state.json
        cron状态 → crontab -l 实时可查
  所以: 我"记错了"不重要, 读文件就对了

误差类型B: "我理解"误差 — 你的需求我理解歪了
  解决: 任务前强制写规划文件(six-page-plan), 把"你要什么"写成文件
        P1-3的LayeredPlanner: L1目标层定义了成功标准
        P2-2的DoD清单: 交付前逐条检查
  所以: 每一步都在对照原始目标, 不是凭记忆跑

误差类型C: "我简化"误差 — 中间结果我简化汇报
  解决: tool_unloader + handoff_notes
        大结果外置: 不是"我简化告诉你", 是"你要看完整结果我read_file"
        交接笔记: 最近10步+最近5条决策, 关键细节不会被简化掉
  所以: 简化的是汇报格式, 不是存储格式
```

## 二、长程任务执行保障（基于Reflexion + 检查点）

### 🔴 已犯2次的坑：ComfyUI方法签名

备份恢复后，`get_prompt(self, kwargs)` 和 `IS_CHANGED(cls, kwargs)` 会被恢复成旧版位置参数。
ComfyUI把所有widget参数作为命名参数传入，位置参数会导致：
`TypeError: xxx() got an unexpected keyword argument '文件夹路径'`

**验证命令（每次备份恢复后必做）：**
```bash
grep -n 'def get_prompt\|def IS_CHANGED' __init__.py
```
两行都必须是 `**kwargs` 而不是 `kwargs`。

### ⚠️ 部署后必须验证cron实际运行

本会话发现了致命模式：**6个关键cron job被self_evolve auto-paused但无人发现**。
context_packer/surgical_slicer/gear_enforcer/hy-memory全部静默停止运行。

**部署新脚本后必须：**

import sys,json
d=json.load(sys.stdin)
paused=[j for j in d['jobs'] if not j['enabled']]
if paused:

### 0.7 修复后的持续验证与退化检测（2026-06-01 全场对话固化）

**🔴 核心教训：修好了不等于永远好了。** 系统退化会在几天/几周内悄无声息地发生。

**修复后的验证必须4层穿透：**
```
层级1: 文件存在          → stat -c%s <file> 确认>100B非空
层级2: 自动运行(cron)    → crontab -l | grep <name> 确认在列表
层级3: 输出数据正确     → 执行脚本/读输出文件，确认非空、非0、内容合理
层级4: 对话层生效        → 系统真的在用这个输出，而不是文件写在那没人读
```

**必须主动检查，不能等人问（2026-06-01 强制规则）：**
执行系统级任务时（修复、增强、部署、升级），**必须主动检查所有子系统的实际运行状态**并输出验证结果。
检查清单：齿轮健康、cron条目、关键文件完整性、输出数据新鲜度、上下文索引可追溯性。

**退化模式清单（已发现）：**
1. cron脚本输出格式与下游消费者预期不一致 → cron每跑一次就覆盖一次，数据越来越空
   - 例：context_index_system.py输出的JSON没有sections字段，consistency_guard读不到就报警
   - 修复：让输出脚本和消费脚本的schema对齐，而非只修消费端
2. 新写的脚本没有被挂入cron → 文件存在、功能正常，但没有自动运行
   - 修复：deploy.py自动挂cron + consistency_guard检查cron条目
3. sqlite3文件被cron覆盖写 → active_memory.db等数据文件被空/旧数据覆盖
   - 修复：写入前读一次校验行数，异常则跳过

```bash
# 1. 脱敏检查
grep -rn "sk-[a-zA-Z0-9]\{32,\}" . --include="*.py" --include="*.md" --include="*.yaml"
grep -rn "/home/" . --include="*.py" --include="*.md"
grep -rn "172\.31\.32\.1" . --include="*.py" --include="*.md"

# 2. 清理编译缓存
find . -name "__pycache__" -type d -exec rm -rf {} + 2>/dev/null
find . -name "*.pyc" -delete

# 3. 验证deploy.py
python3 -m py_compile deploy.py

# 4. 确认无个人信息
git add . && git diff --cached | grep -E "sk-|/home/|password|api_key:" 
```
