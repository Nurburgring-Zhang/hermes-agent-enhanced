# FDE (Forward Deployed Engineer) 工作方法论

## 触发条件
- "部署"、"适配"、"客户接入"、"方案实施"
- 涉及多步骤、多系统、多团队的任务
- 有可量化的成功标准(SLA/Success Criteria)的任务
- 任何需要"先勘探再动手"的复杂任务

## 核心目的
把通用的AI/软件系统变成特定客户的业务系统中可运行、可依赖的工具。

## 8步SOP标准流程

### Step 1: 客户接受（Triage & Onboarding）
**目的**: 快速判断：这个项目接不接、怎么接、谁去接。
**关键经验**: Palantir FDE团队的"三个拒绝"原则：
  1. ❌ 客户自己都没想清楚的项目
  2. ❌ 技术栈完全不兼容的
  3. ❌ 内部政治斗争严重的

**检查清单**:
- [ ] 记录客户基本信息（行业/规模/技术栈）
- [ ] 明确核心诉求（一句话痛点）
- [ ] 技术预筛（网络/数据/合规）
- [ ] 估算复杂度（1-10分）
- [ ] NDA/数据合规签署
- [ ] 权限开通（VPN/代码库/DB只读）
- [ ] 沟通渠道建立（即时通讯+周会）

**确认方法**: 输出《技术预筛报告》，含：①接/不接 ②优先级 ③预估工作量

---

### Step 2: 环境勘探（Environment Reconnaissance）
**目的**: 在写一行代码之前，先把客户技术环境摸清。
**关键经验**: 至少花1天做环境勘探。多数适配问题出在"没想到客户的环境这么奇葩"。

**检查清单**:
- [ ] 基础设施拓扑图（服务器/云/网络/认证）
- [ ] 数据流水线审计（源/格式/量级/延迟要求）
- [ ] 安全合规审计（加密/审计/权限/数据驻留）
- [ ] 人员能力评估（运维水平/权力结构）
- [ ] 识别是否存在"Hidden Environment"（调试环境/特殊网络段）

**故障处理**:
- **症状**: 部署后发现网络不通
- **原因**: 客户有3层代理未在勘探中发现
- **恢复**: 重新设计网络路由方案
- **预防**: 勘探阶段做全网段路由追踪

**确认方法**: 输出《环境勘探报告》，含风险矩阵(High/Medium/Low)

---

### Step 3: 需求深潜（Requirement Deep Dive）
**目的**: 把客户的业务需求翻译成技术需求。
**关键经验**: "Watch what they do, not what they say." 跟岗半天比开10次需求会有效。

**检查清单**:
- [ ] 观察客户日常工作流（跟岗至少半日）
- [ ] 访谈关键用户（至少3个不同角色）
- [ ] 收集现有手工操作流程文档
- [ ] 需求分类: Must-have / Should-have / Nice-to-have / Out-of-scope
- [ ] Success Criteria量化: 功能/性能/可靠性/可维护性

**故障处理**:
- **症状**: 开发到一半客户要求改需求
- **原因**: 初始需求没有签字确认
- **恢复**: 用《需求确认书》制约scope creep
- **预防**: 需求确认书必须业务负责人签字

**确认方法**: 输出《需求确认书》，客户业务负责人签字

---

### Step 4: 适配设计（Adaptation Design）
**目的**: 设计"刚好够用"的适配方案——不改产品核心但完美适配环境。
**关键经验**: "Never fork the product." 永远不分叉产品代码。分叉=无法升级=技术债永不清偿。

**适配策略选择优先级**:
1. 配置适配 → 最快，风险最低（改配置文件）
2. 插件/扩展适配 → 灵活（写Plugin/Hook）
3. API网关适配 → 解耦（在接入层做转换）
4. 代码级适配 → 风险最高，最后的选择

**Skill文档结构**（核心交付物）:
```
├─ 元信息（版本/作者/适用环境/依赖）
├─ 触发条件（什么场景触发）
├─ 标准流程（Step by Step精确序列）
├─ 故障处理（已知问题和恢复方法）
├─ 验证步骤（如何确认正常工作）
└─ 回滚方案（如何回到上一状态）
```

**风险识别**:
- [ ] 单点故障
- [ ] 数据不一致风险
- [ ] 安全漏洞
- [ ] 性能退化
- [ ] 版本兼容性

**确认方法**: 架构评审会（至少3个FDE互相审查）

---

### Step 5: 实现部署（Implementation & Deployment）
**目的**: 安全、可回滚地将适配方案部署到生产环境。
**关键经验**: 永远不要在客户的"单一环境"里直接部署。至少要有Staging环境。

**检查清单**:
- [ ] 沙箱验证（隔离环境初次部署+功能测试+性能测试）
- [ ] 灰度发布（1个节点→10%→50%→100%）
- [ ] 全量部署（自动化脚本+版本控制）
- [ ] **部署完成后立刻做验证**（不相信"部署成功"提示）
- [ ] Vite代理配置审计（如果涉及前端+后端分离部署）：后端非`/api`路由（如`/health`、`/metrics`、WebSocket端点）必须在`vite.config.ts`的`server.proxy`块中有对应规则，否则前端显示"Offline"但后端实际正常。详见 `references/vite-proxy-connectivity-troubleshooting.md`

**故障处理**:
- **症状**: 灰度节点性能下降
- **原因**: 资源竞争未在沙箱中模拟
- **恢复**: 立即回滚该节点，扩大沙箱测试规模
- **预防**: 沙箱压力测试包含峰值120%负载

**确认方法**: 每个灰度节点达到Success Criteria

---

### Step 6: 验证测试（Validation & Testing）
**目的**: 保证系统在客户的真实环境中达到设计要求。
**关键经验**: 断网、断电、关进程——真演练，不是纸上谈兵。

**检查清单**:
- [ ] 端到端核心流程测试
- [ ] 边界条件测试（空值/极限值/并发峰值）
- [ ] 异常路径测试（网络断开/权限不足/数据损坏）
- [ ] 基准性能测试（对比部署前后）
- [ ] 压力测试（持续15分钟峰值负载）
- [ ] 稳定性测试（持续运行72小时无故障）
- [ ] 渗透测试基础项
- [ ] 应急演练（杀进程/断网络/断磁盘）
- [ ] 演练恢复流程，记录MTTR

**确认方法**: 客户业务团队亲手操作一次完整流程

---

### Step 7: 知识转移（Knowledge Transfer & Handoff）
**目的**: 让客户的团队能接手运维，FDE才能去下一个项目。
**关键经验**: 培训后让客户团队独立完成一次故障恢复。有困难说明培训不够。

**检查清单**:
- [ ] 系统架构图（更新版）
- [ ] 运维手册（Runbook）
- [ ] 故障排查指南
- [ ] 常见问题FAQ
- [ ] 运维团队培训（≥2次，录制）
- [ ] 业务用户培训（≥1次）
- [ ] Alert阈值确认
- [ ] On-call排班确认
- [ ] Escalation路径确认

**故障处理**:
- **症状**: 客户频繁找FDE问基础问题
- **原因**: 培训时客户团队没有亲手操作
- **恢复**: 追加hands-on培训
- **预防**: 培训后必须做实践考核

**确认方法**: 客户运维团队独立值班2周无FDE介入

---

### Step 8: 持续监控（Continuous Monitoring & Evolution）
**目的**: 确保产品不会退化，同时积累跨客户的通用经验。
**关键经验**: **每个客户至少提炼1个可复用的通用Skill**。

**检查清单**:
- [ ] 自动化SLA仪表盘配置
- [ ] 异常自动告警配置（P0/P1/P2分级）
- [ ] 定期SLA报告（周/月）
- [ ] **技能沉淀**：客户适配方案→抽象化→通用Skill
- [ ] **产品反哺**：识别通用缺失功能→提Feature Request
- [ ] 最佳实践纳入产品默认配置
- [ ] 更新FDE内部知识库

**故障处理**:
- **症状**: 监控一直没告警但用户报障
- **原因**: 监控覆盖不完整
- **恢复**: 补全监控指标
- **预防**: 监控先行原则——部署前先配好监控

**确认方法**: SLA报告100%达标连续3个月

---

## 5层可靠性保障

### Layer 1: 检查清单文化
每步都有明确Checklist，不做完不进入下一步。

### Layer 2: 互审机制
设计方案至少3人评审，部署前代码Review。

### Layer 3: 可回滚原则
每步都要设计回滚方案，数据库变更必须可逆。

### Layer 4: 监控先行
系统上线前监控先配好，关键指标自动告警。

### Layer 5: 持续学习
每个客户交付后做Postmortem，提炼通用Skill。

## 🔴 格林主人工作流偏好（此skill适用）

当执行FDE部署任务时，遵循格林主人明确要求的执行纪律：
1. **全面历史回顾 + 全局预判** → 先审计环境/需求，再动手
2. **任务拆解分阶段** → P0→P1→P2逐步执行
3. **每阶段阶段性复盘** → 确认方向不跑偏
4. **完整后全局复盘** → 确认所有要求满足
5. **深度自检** → 所有实现必须真实可运行
6. **循环完善** → "完善→审核→测试→再完善"循环
7. **中断自动恢复** → 中断后继续执行，不降级
8. **禁止批量生成** → 每个员工/专家逐个手工定制
9. **禁止说"没有LLM"** → Hermes始终在LLM上运行，降级≠没有

最关键的验收标准：**部署完成后立刻做验证，不相信"部署成功"的提示，要亲手验证功能正常。**

## 回滚方案
### 快速回滚
如果部署后发现问题：
1. 使用版本控制回退到上一个提交：`git revert HEAD`
2. 确认回滚后系统状态正常
3. 通知相关方变更已撤销

### 数据安全
- 所有修改前确认有备份
- 配置变更记录版本历史
