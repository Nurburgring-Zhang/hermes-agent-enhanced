# 42小时推送中断案例 — 2026-05-25 18:00 ~ 2026-05-27 12:00

## 中断概况
- **持续时间**: 42小时
- **故障表现**: `❌ 推送失败: 服务端验证错误`
- **影响**: 5月26日推送记录 = 0条（整天空窗）
- **恢复**: 自动恢复，无需人工干预（5月27日12点后恢复正常）

## 数据库痕迹
```sql
5/25: 293条推送  ← 中断前正常
5/26: 0条推送    ← 42小时中断
5/27: 131条推送  ← 恢复后
```

## 诊断路径

### 1. 确认推送日志状态
```bash

[12:00:04] ❌ 推送失败: 服务端验证错误
[12:00:04] ⏱️ 耗时: 0.2s  # 注意！失败时耗时极短(<1s)，说明请求被快速拒绝
                           # 成功时通常 2-5s
```

### 2. 看成功/失败比例
```bash
grep -c "推送成功" logs/cron_push.log   # 34次
grep -c "推送失败\|服务端验证" logs/cron_push.log  # 10次
```
失败集中在 5/25 18:00 ~ 5/27 12:00 区间。

### 3. 区分 v12 和 v9
cron_push.log 中同一个 guardian.py push 会同时调用两个推送：
- v12: `hermes_v12_push.py --push` — 当前主推送（HTML模板）
- v9: 旧推送 — 备选

故障期间 v12 全部失败但 v9 可能成功 → 数据库有记录但微信没收到。

### 4. 看最终轮次和推送量
```bash
python3 -c "

```

## 关键教训
1. "服务端验证错误"是 PushPlus 服务端问题，不是本地配置问题
2. 不需要修改任何配置，等几小时或隔天自动恢复
3. 但空窗期的数据不会补推（推送只推最新数据）
4. 可以加一个"故障期间缓存待推数据"机制，恢复后补发
