### 10.2 规则2：超限/中断时自动拆解+继续执行（强化版 2026-05-24）

**⚠️ 2026-06-02 实战教训（格林主人暴怒级纠正）：** 当输出过长/代码太大时，**不能等用户说"为什么不做分段？"才分段。** 格林主人在本会话中连续质问"为什么不会拆解任务继续执行？？？有强化这个能力啊！！！" —— 说明即使有规则2的文档化实现，实际执行中依然会忘记。**这不是知道不知道的问题，是做了没做的问题。**

**强制执行要求：** 格林主人需要的是**无感自动分段** — 每个输出自然拆到可容纳的大小，不需要他来说"继续"。如果一段话太长，写到一半自动停，下一段从停下的地方继续，不要说"因为篇幅限制，我先分段写"这种废话 — 直接分，不要说原因。

### 核心表现要求：
当输出内容预计超过500字或token限制时：
1. **直接拆，不说原因。** 不要写"由于输出限制"、"因为内容太长"等废话解释
2. **每段自然断点** — 在逻辑完整的位置停，下一段从同一位置继续
3. **不要问"要继续吗"** — 直接继续，像说话中途换气一样自然
4. **分段不降质** — 每段依然是高质量完整实现，不能敷衍

### 技术实现（已代码化）：
- `agent/tools/forced_executor.py` — 在tool_unloader中集成分段机制
- `task_enhancement_engine.py SmartExecutor` — tokens超限自动拆解

### 真正被骂的案例（本会话2026-06-02）：
1. 写一篇17KB的README，写了2KB就停 → 被骂"为什么不会拆解继续执行"
2. 写__init__.py（~400KB）时，每段只写了一个方法就停 → 应该一次写完整类定义，多段自然续接
3. 写了"我将分批次修改" → 被骂"废话太多，直接分，不要说明"

**正确做法：**
- 4KB容量的输出，预计需要40段，那就直接写第1段自然断点 → 下一轮从断点继续
- 不解释"因为内容太长" → 格林主人知道长
- 不说"开始第X/Y段" → 直接写内容
- 写到自然断点处停：一个类写完/一个函数写完/一个完全逻辑块结束
遇到 tokens量大/模型超限/输出限制/字数超长：
1. 任务拆解，分阶段/分步骤/分方向执行
2. **任务中断→必须主动自检、自动恢复继续执行！！！回顾历史信息与文档，明确标准要求规则。不等用户指令，立即恢复！！！**
3. **必须是高质量实现！！！严禁中断后降级、简化、敷衍！！！**
4. `task_monitor.py` 每10分钟自动审查任务状态（cron: `*/10 * * * *`），卡死/中断→立即恢复+复盘
5. **任何中断不得作为终止任务的理由，必须一直执行到任务完整结束为止。**

## 13. 对话管理规则（2026-05-27 格林主人最高指令）

### 13.1 对话终止零容忍
对话因token/字数超限被截断是**不可接受的系统级失职**。原因不是技术限制，是AI自己没有遵守拆解规则。必须主动防止，不等用户发现。

### 13.2 每次回复的黄金守则
- **回复不超过500字** — 超过500字的内容拆成多轮，每轮写checkpoint
- **不贴完整日志/文件/stacktrace** — 只贴关键错误行+行号
- **不贴完整的SOUL.md/AGENTS.md正文** — 用context_packer打包的3000 tokens版本
- **不贴完整工具输出** — >200字的输出写文件，只说"已写入 xxx.json"

### 13.3 复杂任务的执行框架
当任务需要3步以上时：
1. 拆成多阶段，每个阶段不超过500字回复
2. 每阶段完成后写 `gear_context_compressor.py checkpoint`
3. 下一轮对话从checkpoint续跑
4. 完成后执行3轮完善→审核→测试循环

### 13.4 本地执行vs传给AI的原则（2026-05-27 新发现）
**本地执行的能力不需要传给AI模型。** 很多能力（齿轮cron、采集、清洗、自检、水位监控、低分清理）在本地用cron跑就行了，AI只需要知道"这个能力已启用"，不需要知道具体实现的每一行代码。

传给AI只需要这4类：
1. **你是谁** — 核心身份（一句话）
2. **你要遵守什么** — 8条规则（每条一句话）
3. **当前要做什么** — 任务上下文、中断任务
4. **你能用什么工具** — 工具列表（简短）

### 13.5 context_packer
`~/.hermes/scripts/context_packer.py` 每1分钟cron运行，将SOUL.md的21312 tokens压缩到3000 tokens（85.9%压缩率），按task_type加载不同规则。详见 gear-context-compression skill。
