## 核心质量框架：7维度自然融合

每条高质量提示词必须包含以下7个维度，**自然融入一段500-800字的完整中文文字**，不要标题、不要序号、不要分段：

| # | 维度 | 内容要点 | 篇幅参考 |
|---|------|----------|---------|
| 1 | **风格技法** | 开篇明确风格名称+技法特征细节（笔触、颜料、画布材质、摄影手法、数字渲染特性） | 2-4句 |
| 2 | **镜头构图** | 景别（特写/近景/中景/全景）、视角（平视/仰视/俯视）、构图法则（三分法/对称/引导线/框架式/黄金分割/留白） | 1-2句 |
| 3 | **主体姿势动作** | 精确位置（左/右/中）、身体姿态（站立/坐/蹲/躺）、头部角度、眼神方向、表情细节（嘴角/眉梢/瞳孔）、手臂和手的细节（手指弯曲/握拳力度/指尖轻触）、腿部姿势 | 5-8句（核心区） |
| 4 | **材质细节** | **头发**（颜色光泽/长度发型/纹理/光照效果）、**皮肤**（肤色/肤质/面部特征/斑痕/妆容/血管）、**服装**（面料名称/颜色图案/具体设计/物理特性/穿着状态）、**褶皱磨损**（类型/分布/磨损痕迹/老化/污渍）、**配饰**（首饰/手表/眼镜/帽子/围巾/包袋/鞋/手套） | 8-12句（最密集区） |
| 5 | **前中后景** | 前景（引导视线/制造景深/质感最真实）、中景（主体活动空间/与周围元素关系）、背景（交代时间地点/烘托氛围/提供色彩对比） | 3-5句 |
| 6 | **色彩系统** | 主色+辅色+对比色+冷暖倾向。**必须使用中国传统色名**（红：朱红/胭脂/银红/绛紫/绯红/赭色；橙黄：琥珀/橘黄/藤黄/鹅黄/牙色/金色；绿：石绿/翠绿/竹青/苍绿/黛绿/碧色；蓝：群青/靛蓝/钴蓝/黛蓝/天青/月白；紫：藕荷/丁香/紫檀/玫瑰紫/雪青；棕褐：赭石/檀褐/栗色/茶色/古铜/驼色；黑白灰：墨黑/象牙白/银灰/烟灰/骨白/米色/霜色） | 3-4句 |
| 7 | **光影氛围** | 光源方向（顺光/侧光/侧逆光/逆光/顶光/底光/漫射光）、光线性质（硬光/柔光/散射光）、高光描述（位置/形态/亮度/色彩）、阴影描述（位置/颜色/边缘/层次）、轮廓光/边缘光。**氛围情感收尾**（宁静/温暖/忧伤/神秘/激昂/慵懒等） | 4-6句光影+2-3句氛围 |

## 字数铁律

- **每条严格500-800中文汉字**（可略微浮动到750字左右最优）
- 超过800字必须缩减：删除冗余修饰语、合并短句、压缩次要元素
- 低于500字必须补充：增加材质细节、层次描述、光影特征
- **库文件条目严格100-200字**：风格库/穿搭库/姿势库每条精简描述，不超200字

## 格式禁令

- ❌ 不要标题、不要序号、不要分段
- ❌ 禁止科技词汇（量子、夸克、粒子、齿轮、青铜、全息、赛博等）
- ❌ 禁止数字化描述（45度、3厘米、5毫米、2点钟方向等具体数值）
- ❌ 不要模板化开头（如"一幅xxx风格的画作"），直接开始描述
- ❌ 每条必须独一无二，禁止重复之前的场景/姿势组合
- ✅ 一条就是一段完整的文字，可以有逗号和句号
- ✅ 每行一条，纯文本

## 大规模生成策略（1000+条）

当需要生成大量提示词（如10万条级别）时：

### 策略1：九层模板法（高质量单条生成）
每一条按照九层结构逐层构建，最终写成一段完整文字：

| 层 | 内容 | 字数占比 |
|----|------|---------|
| 1 | **风格技法**（开篇）：风格名+技法特征+光线氛围 | 10% |
| 2 | **镜头构图**：景别+视角+构图法则 | 5% |
| 3 | **主体姿势**：位置+姿态+头角度+眼神+嘴角+手指+腿 | 20% |
| 4 | **材质细节**（核心区）：头发+皮肤+服装面料+配饰 | 30% |
| 5 | **前中后景**：前景引导→中景活动→背景烘托 | 10% |
| 6 | **色彩系统**：主色+辅色+对比，用传统色名 | 10% |
| 7 | **光影系统**：光源方向+软硬+高光+阴影+轮廓 | 10% |
| 8 | **氛围情感**（收尾）：整体氛围+主体状态 | 3% |
| 9 | **出圈要素**（画龙点睛）：风格致敬/悖论/反差/错位 | 2% |

### 策略2：delegate_task子代理（适合少量高质）
- 每个子代理每次只生成5-10条（因迭代次数限制）
- 每生成1条立即用 `echo "内容" >> 文件` 追加写入
- 生成后用 `echo "内容" | wc -c` 验证字数
- 超800字立即缩减，低500字立即补充
- 每批3个子代理并行，产出约24-30条
- **⚠ 注意**：delegate_task受父对话超时影响，如果父对话有间隔中断，子代理会被杀死导致无产出。需要频繁轮替或改用cronjob。

### 策略3：cronjob后台持续生产（适合中等规模、有cron环境）
- 创建多个cronjob：`cronjob action=create schedule="every 15m" repeat=1000`
- 每个负责一个文件，每次生成5-8条
- 写入前先 `wc -l` 检查是否已达目标（如11111条）
- **每1分钟运行一次**的cronjob：9个文件×每个5条=45条/分钟=2700条/小时
- 不受对话超时影响，持续后台运行
- 写入后输出标记（如"pg2ok"）便于追踪

### 策略4：库+组合（适合模板化小批量）
- 建立姿势库、穿搭库、服饰库、鞋帽库、环境库、物品库（每库3000条）
- 从各库随机选取组合，AI润色为完整一段

### ✅ 策略5：Python批量组合引擎（推荐——最快，10万+条级别首选）

当需要真正大规模产出（1万-20万条）时，**不要用AI逐条生成**，因为AI每条的生成速度（~30秒/条）根本无法接受。改用Python脚本从模板库随机组合：

#### 原理
从2,000条风格描述库 + 2,000条穿搭库 + 2,000条姿势库 + 127条场景库 + 20种开头模板中，用Python `random.choice` 排列组合，每组合一次=生成1条新的唯一prompt。**0.002秒/条**，比AI快15,000倍。

#### 脚本核心结构
```python
# 加载模板库（每库2000+条）
styles = load_lines("15_女性风格库.txt")      # 2000条
outfits = load_lines("16_女性穿搭库.txt")      # 2000条
poses = load_lines("17_女性姿势库.txt")         # 2000条
scenes = load_lines("13_场景库.txt")            # 127条

# 几个关键库（必须在生成前准备好，每条都在100-200字之间）
# 15_女性风格库.txt — 风格描述段落
# 16_女性穿搭库.txt — 穿搭描述（含传统色名）
# 17_女性姿势库.txt — 姿势+动作描述
# 13_场景库.txt — 日常生活场景描述
# 18_摄影风格库.txt — 摄影风格/大师名作描述（148条）

def gen():
    style = random.choice(styles)
    scene = random.choice(scenes)
    outfit = random.choice(outfits)
    pose = random.choice(poses)
    c1, c2 = random.sample(COLORS, 2)
    
    # 20种开头随机切换，避免前20字重复
    r = random.random()
    if r < 0.2:   opening = f"在{style[:40]}中，{scene[:60]}"
    elif r < 0.4: opening = f"{scene[:60]}被{style[:40]}的光影包裹"
    # ...
    
    line2 = f"镜头采用{shot}的{comp}捕捉，浅景深模糊背景..."
    line3 = f"穿搭以{c1}与{c2}为主调，{outfit}。"
    # ... 逐层拼接
    
    return opening + "，" + line2 + line3 + ...  # 最终550-850字
```

#### 去重策略（关键）
```python
# 使用前20-30字做key即可（多层模板自然保证开头多样性）
existing = set()
key = prompt[:30]
if key not in existing:
    existing.add(key)
    write(prompt)
```
- 每个文件独立去重（不跨文件）
- 用前20-30字做key，因为prompt的xx风格开头已足够区分
- 每小时可产出5,000-10,000条

### 🔥 格林主人专用：库级Prompt工程 → 维度库架构 + 组合引擎 + 质量循环

当用户要求"建立维度库"或"清洗源文件"或"建立角色特征库/场景库/穿搭库"时，触发的不是单条prompt生成，而是整套**维度库架构 + 组合引擎**的建设。这个架构比单条prompt生成更复杂，需要严格方法。

#### 核心架构：15维度库体系（2026-05-23完整实现）

所有15个库已建立，路径、条数和标签格式详见 `references/dimension-library-architecture.md`。

**建立/扩展库时要严格遵守的规则：**
1. 每条标注标签前缀（如 `[视角:居中构图]`），用于组合校验
2. 分布比例必须**匹配to_chi全量数据的真实分布**，不能随意设定
3. 每条**禁止比喻**（仿佛、犹如、就像、好似、宛如、如同）
4. 每条**禁止面部红润/潮红/泛红/雀斑/泪痣**
5. 每条**有具体画面感**，不是抽象概念
6. 涉及身体部位必须**左右区分**（左手/右手、左臂/右臂）

#### 组合引擎：从各维度库智能融合生成完整prompt

完整prompt生成步骤（见 `/tmp/composer_v7.py`）：
1. 从场景库选一个场景
2. 根据场景名从匹配规则表查合法穿搭+姿势类型
3. 从穿搭库和姿势库中筛选与合法类型匹配的条目
4. 从其他维度库随机抽取
5. 按"场景→角色出场→穿搭姿势→光影→色彩→氛围"顺序自然融合

#### 质量循环：生成→对比→修正→再生成

建立维度库后必须通过质量循环验证：
1. 生成10条组合prompt
2. 与参考文件（`小红书 （10).txt`）对比每条的长度、左右区分、场景匹配度
3. 如果质量不如参考文件，找出具体缺陷：
   - 场景-穿搭-姿势不匹配 → 修正匹配规则表
   - 左右区分不够 → 修正库条目
   - 长度不足 → 补充细节
4. 修正后重新生成对比
5. **循环直到生成质量全面超越参考文件**

**关键质量指标：**
- 组合引擎对比测试胜率 ≥80%（15维度组合理应100%）
- 每条 ≥250字（参考文件平均287字）
- 90%+条有"左/右"区分（参考文件87%）
- 场景标签命中率100%
- 无比喻词、无面部红润词

#### ⚠️ 重要：建立库时优先从数据源提取而非从头生成

建立维度库时，应**优先从现有高质量prompt数据源中提取真实句子**，而不是从头AI生成。数据源质量排序：

### 反模式1：三手矛盾（多手问题）

组合prompt时最常见的严重错误：

**错误示例（不可接受）：**
```
她正在用锅铲翻动食材。一只手自然垂落，另一只手轻触一扇老旧的木窗的边缘。
```
→ 画面上同时存在三只手：一只手拿锅铲、一只手垂落、一只手轻触窗户。

**正确做法：**
- 一个人只有两只手，只能用一次"左手XX，右手XX"来描述
- 如果主体在"正在做某事"，就不能同时描述"一只手自然垂落"
- 禁止句式库：`一只手自然垂落`、`另一只手轻触`、`轻触XXX的边缘`

**根源分析：** 库条目本身可能包含`特写照片显示一个人...一只手放在...另一只手...`，组合引擎抽到这条后又在prompt里加了"她正在做..."，就变成三手。

**修复策略：**
1. 库条目中保留精确手部描述（如`左手放在膝盖上，右手端着茶杯`）——这只有两只手，没问题
2. 组合引擎中，如果角色库条目已经包含了手部动作（检测到"左/右手"关键词），就不要再从活动库抽取"正在做"的活动
3. 或者在抽取时检查：如果角色库条目的手部描述+"正在"活动 = 超过两个手部动词，则丢弃活动描述

### 反模式2：多角色拼接

组合prompt时，同一个prompt中出现了两个不同的角色描述：

**错误示例（不可接受）：**
```
一位年轻亚裔女性坐在教室课桌前，留着黑色长直发...一位留着齐肩棕色短发的年轻女性站在阳台上晒太阳...
```
→ 一张画里有两个人，但组合引擎没有交代他们之间的关系。

**根源分析：** 角色库的每条条目都以`一位[性别]`开头，组合引擎从01库抽了一条角色，又从活动库抽了一条活动描述（活动库条目也可能包含`一位年轻女性`），两条拼接在一起形成多角色。

**修复策略：**
1. 角色描述在加载时去掉`一位/一名`前缀（`re.sub(r'^一位|^一名', '', content)`）
2. 从01库只抽一次角色，整条prompt只用这个角色描述
3. 活动库抽取活动时，只取"正在XXX"部分，丢弃活动描述中的人物部分
4. 最终组合时：风格+场景+角色(一次)+活动(一次)+其他维度

### 反模式3：裸体时的服装描述

当角色描述中出现`裸体`/`赤裸`/`裸露`等词时，必须跳过服装材质和配饰描述。

**原因：** 裸体状态下写"她身穿的面料呈现出丝绸般的光泽"明显矛盾。

**检测方式：**
```python
is_nude = any(w in role_text for w in ['裸体','赤裸','裸露','一丝不挂','未着寸缕'])
if is_nude:
    # 跳过材质、配饰、服装相关描述
    material = ''
    accessory = ''
```

### 反模式4："用XX剪裁而成的衣物"硬拼接

材质库中的条目直接硬塞进了服装描述，形成病句：

**错误示例（不可接受）：**
```
用皮革表面布满天然的毛孔纹理，边缘处有轻微的使用磨损痕迹剪裁而成的衣物随身体的姿态自然垂落
```

**修复方式：** 材质描述要用自然语言描述人物穿着的服装质感，不是把材质描述原文贴在服装位置。正确做法：从材质库提取关键质感词，然后用自然语言构建"她身穿的上衣表面有XXX质感"。

### ⚠️ 维度使用策略：不是每个维度都必须用

用户明确要求："不是每个维度都用上，在合理的前提下，多使用维度"。

**规则：**
- 不是每条prompt都必须硬塞全部15个维度
- 合理的前提下尽可能多用——但不要为了凑维度而写矛盾/冗余的描述
- 例如：如果角色是"裸体"，就不能硬塞服装材质描述（11库）和配饰描述（12库）
- 例如：如果角色正在做某件事（活动库），就不要同时描述"一只手自然垂落"的静态姿势
- **优先级**：流畅自然 > 维度覆盖 > 字数达标

### ⚠️ 裸体时的服装规则

当角色描述中出现以下词时：
```python
is_nude = any(w in role_text for w in ['裸体','赤裸','裸露','一丝不挂','未着寸缕'])
```

**规则：**
- ✅ **保留**裸体/赤裸描述（用户明确要求）
- ✅ **跳过**服装材质描述（11_材质质感库）
- ✅ **跳过**配饰描述（12_配饰鞋帽库）
- ✅ **保留**其他所有维度（色彩、光影、情感、场景等）
- ❌ 不要修改库文件中的"裸体"词——保持原始数据

### ⚠️ "正在"活动时的服装描述

用户明确要求：**有"正在"活动时，不需要去掉服装描述**。

这与裸体规则不同。当角色正在做某事（煮饭、看书、走路），服装描述仍然合理且应该保留。只有裸体时跳过服装。

### ⚠️ 库文件损坏风险

在批量处理过程中，以下操作可能导致库文件损坏：
1. **正则替换把内容删空了** — 如12_配饰鞋帽库.txt被替换规则清空为0条
2. **去重过度** — 如14_色彩搭配库.txt去重后从200条变成20条

**预防措施：**
- 执行批量替换前先备份：`cp 12_配饰鞋帽库.txt 12_配饰鞋帽库.txt.bak`
- 替换后用 `wc -l` 检查文件行数是否合理
- 如果库文件损坏，从原始数据源重新提取：`to_chi/javrate/` 和 `to_chi/xhs/`
- 12库恢复命令：从to_chi中提取配饰关键词匹配的条目，分类整理写入
- 14库恢复命令：从预设的20种色彩搭配描述中随机重复到200条

### 反模式5：固定场景模板重复

```
3. 检查：风格开篇 ✓ 7维度完整 ✓ 无矛盾 ✓ 无比喻 ✓ 无科技 ✓ 字数达标 ✓
4. 输出一条完整文字

【已生成列表】（子代理内部自动维护，用前30字做key）
```

### ⚠️ 关键陷阱：子代理间去重

10个子代理分别生成，各自都不知道别的子代理生成的内容。跨文件去重需要在汇总阶段做：
- 使用 `simhash` 或 `textdistance` 做相似度检测
- 阈值：Jaccard相似度 > 0.7 视为重复
- 保留较早生成的那条，删除较晚的

### ⚠️ 关键陷阱：delegate_task超时

delegate_task有超时限制，逐条生成1000条会在中途超时中断。
**修复策略：**
- 每个子代理先运行一次生成框架（定义风格+场景+配色方案），一次性写入框架文件
- 然后分批生成：每次delegate_task生产20-30条 → 写入txt → 继续生产下一批
- 使用 `wake_guide.json` 或文件行数追踪进度
- 子代理崩溃后，读取文件行数，从断点继续

## ⚠️ 关键陷阱：格式不是通用的——不同模型的文本编码器差异

### 核心认知：不要用SD认知套非SD模型

| 错误认知 | 正确认知 |
|---------|---------|
| "所有模型都需要质量词 masterpiece" | Qwen架构和Mistral架构不需要，masterpiece对它们只是字面词 |
| "所有模型都理解()加重语法" | Qwen和Mistral不解析()加重，视为普通括号字符 |
| "所有模型都要纯英文prompt" | Z-Image/Qwen-Image原生支持中文，比英文效果更好 |
| "所有模型都用同一个负面词词库" | 非SD模型需要精简负面词（ugly/deformed等标记无效） |

### 模型自适应提示词格式

当为不同模型撰写prompt时：

**为Z-Image / Qwen-Image（Qwen架构）写prompt：**
- ✅ 直接用中文自然语言描述，保留中英文混合
- ✅ 用完整句子而非tag式堆叠
- ❌ 不要加masterpiece/best quality
- ❌ 不要用(())、[]、{}等标记
- 示例: "一位优雅的女性站在樱花树下，微风拂起她的长发，花瓣飘落在她的肩头"

**为FLUX.2 Dev / Klein（Mistral架构）写prompt：**
- ✅ 纯英文自然语言描述
- ✅ 简洁，避免过度长句
- ❌ 不要加masterpiece/cinematic等质量词
- ❌ 不要用SD格式标记
- ❌ 不要有中文
- 示例: "A woman standing under cherry blossom trees, wind blowing her hair, petals falling on her shoulders"

**为SD1.5/SDXL/SD3/FLUX.1（CLIP架构）写prompt：**
- ✅ 纯英文逗号分隔tag式
- ✅ 可以加masterpiece, best quality
- ✅ 可以使用(())加重
- ❌ 不要有中文

## 注意事项和陷阱

### 常见错误
1. **字数超标**：首次生成往往超过800字（经常到1000-2500字），必须主动裁剪。策略：去掉冗余的形容词组合、合并并列小句、简化次要物体描述
2. **字数不足**：低于500字的提示词缺乏细节密度，需要补充材质细节（头发纹理、皮肤质感、服装褶皱）
3. **模板化开头**：避免"一幅xxx风格的画作"，直接开始"xxx风格的画面中..."
4. **缺乏7维度完整性**：最常见遗漏的是色彩系统（忘记用传统色名）和光影细节
5. **写入文件失败**：使用 `echo >>` 时注意中文字符兼容性，推荐用 heredoc 方式

### 写入文件可靠方式
```bash
cat >> /path/to/file.txt << 'EOF'
完整的提示词内容（500-800字中文）...
EOF
```

### 验证字数
```bash
echo "内容" | wc -c     # 总字节数
echo "内容" | wc -m     # 总字符数（中文按实际字数算）
```
