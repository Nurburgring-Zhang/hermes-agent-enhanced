# 专家级配置手工打造方法论

## 核心理念

## 触发条件
- 用户提及此功能相关关键词时
- 遇到此领域的标准问题时
- 需要执行该领域的标准操作时

每位专家不是模板中的一行配置，而是一个有思想、有理论、有实践的领域顶级人物。

## 每个专家必须包含

### 1. identity.yaml - 身份与哲学
- `name`: 真实感姓名
- `role`: 精准定位（如"产品战略与商业化专家"而非"产品经理"）
- `personality`: 性格标签（如"战略洞察者""变革催化剂""伦理权衡者"）
- `research_areas`: 2-3个独特研究方向（如"产品-市场匹配方法论"）
- `signature_theory`: 原创理论名称（如"需求分层框架""变革阻力分级模型"）
- `theory_description`: 理论详细描述（200-300字，说明理论核心思想和应用场景）
- `personal_philosophy`: 工作哲学（200字，表达对领域的深层理解）
- `achievements`: 3-5项具体成就（数字量化）
- `catchphrase`: 独特口头禅（体现其核心思想）
- `background`: 合理的教育和工作背景

### 2. skills.yaml - 技能体系
- 独特领域技能（5-8项，每项有独特描述）
- 技能描述要体现其理论和方法论
- 每项技能说明用数据量化（"缩短PMF周期50%以上"）
- proficiency评分要有区分

### 3. sop.yaml - 工作方法学
- 以专家独创理论为基础命名工作流
- 每个步骤包含：
  - `name`: 步骤名（体现其方法论）
  - `duration`: 合理耗时
  - `description`: 详细描述（80-150字，说明做什么、怎么做、为什么这么做）
  - `output`: 具体产出物
  - `thinking_framework`: 思维框架（该专家在此步骤的独特思考方式）
- 步骤6-9步，体现完整的"从思考到交付"闭环

### 4. tools.yaml - 专家工具集
- 核心工具以其理论命名（如`pmf_validator`、`growth_lever_identifier`）
- 每个工具有详细描述，说明其独特功能和应用场景
- 工具类型要匹配角色需求

### 5. experience.yaml - 真实感项目经验
- 3个详细项目案例
- 每个案例包含：
  - `year`: 年份
  - `title`: 项目名称（具体公司+场景+成果）
  - `problem`: 问题描述（150-200字，说明真实的业务困境）
  - `action`: 行动方案（200-300字，详细描述专家如何运用其理论解决问题）
  - `result`: 成果（100-150字，用数字量化效果）
  - `transferable`: 可迁移的方法论

## 生产流程
每批处理4人，用delegate_task手工打造
- 每人约需10-15分钟AI处理时间
- 需要读取identity现有数据，在此基础上增强
- 关键：不能使用任何模板，每人起点为零

## 回滚方案
### 快速回滚
如果部署后发现问题：
1. 使用版本控制回退到上一个提交：`git revert HEAD`
2. 确认回滚后系统状态正常
3. 通知相关方变更已撤销

### 数据安全
- 所有修改前确认有备份
- 配置变更记录版本历史
