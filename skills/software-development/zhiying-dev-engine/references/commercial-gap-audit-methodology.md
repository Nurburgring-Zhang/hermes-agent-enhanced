# 商用级差距审计方法论
## 基于2026-06-12 Hermes Agent全系统极端深度审计实战

## 审计范围
全系统扫描294个.py文件、180个skills、66+ cron任务、13个插件、5个修改的官方文件

## 审计流程（7步）

### Step 1: 三路信息收集
- session_search（历史会话回顾）
- fact_store（事实库经验教训）
- 文件系统扫描（目录结构+文件计数+代码行统计）

### Step 2: 分类扫描
按以下15类问题逐类扫描，每类至少找3个实例：
1. 死函数/死模块 — 存在但从未被调用
2. 壳子实现 — 声明了功能但只有mock/占位符
3. 基础设施缺失 — logging/rate_limiting/auth
4. 硬编码凭据 — API密钥/token/密码明文存储
5. 零单元测试 — 无测试文件/无tests目录
6. 异常处理问题 — bare `except: pass` 吞异常
7. 类型签名不匹配 — 函数标注与实际返回值不一致
8. 路径硬编码 — `Path("/home/user/...")` 非 `Path.home()`
9. 线程安全问题 — 全局变量在多线程下竞争
10. 资源泄漏 — 未关闭的文件句柄/SQLite连接/线程池
11. 脚本存在但cron未注册
12. cron条目引用了不存在的脚本
13. 功能重复 — 两个或多个模块做同一件事
14. 全局变量污染 — 跨函数共享可变状态
15. 无版本标记 — 无 `__version__`/pyproject.toml/requirements.txt

### Step 3: 三AI去重
三个独立Agent（架构/安全/工程视角）各自输出报告，交叉对比：
- 发现重复计数项（同一个问题被不同Agent重复检出）→ 合并
- 发现某类问题被所有Agent遗漏 → 补审
- 修正优先级（P0/P1/P2）不一致 → 以最高级别为准

### Step 4: 严重级别标定（7个P0标准）
- API密钥明文 → P0（资产损失风险）
- SQL注入 → P0（数据安全底线）
- 壳子实现（架构核心模块）→ P0（声称为功能实则从未运行）
- bare `except: pass` 在守护进程模块 → P0（Ctrl+C无法中断）
- 零认证 → P0（外部可访问服务无防护）
- 启动脚本引用不存在服务 → P0（系统无法正常启动）
- 无审计日志 → P0（安全事件不可追溯）

### Step 5: 修复规划（并行+串行混合）
```
Phase-0: 紧急止血（API密钥迁移+ .gitignore+ .env安全加载器）
Phase-1: 安全基础设施（SQL注入修复+认证系统+审计日志）
Phase-2: 架构修复（六部接口契约+死代码清理+壳子真实实现）
Phase-3: 异常处理重构+集中配置
Phase-4: 测试基础设施
Phase-5: logging+速率限制+可观测性
Phase-6: CI/CD+静态分析
Phase-7: 技术债务清理
```

### Step 6: Phase-0执行模板
Phase-0 (紧急止血)的标准步骤：
1. 创建.gitignore（阻止密钥/DB/缓存被提交）
2. 扫描所有配置文件提取API密钥
3. 创建env_loader.py（自动加载.env+支持`${ENV_VAR}`解析）
4. 将密钥从config.yaml迁移到.env，替换为`${ENV_NAME}`引用
5. 在model_tools.py启动加载中加入env_loader
6. 设置.env权限为600
7. 验证所有provider密钥是否从环境变量成功解析
8. 确认所有API endpoint使用HTTPS

### Step 7: 交付验收
- 每个阶段必须有实际运行证据（curl/pytest/浏览器截图）
- 三AI互审交叉验证验收结果
- 审计报告保存到 `/mnt/d/Hermes/备份/` 目录
