# 方案v3批量补齐模式 (2026-06-16实战固化)

## 场景
有完整的产品设计文档（如36卷方案v3），需要对照当前项目找出缺失功能并分批补齐。

## 流程

### Step 1: 研读方案
- 总索引 → 产品愿景 → 功能清单PRD(如34_功能模块清单PRD.md) → 路线图
- 输出：M0-M6各阶段功能清单

### Step 2: 差距对比
按功能模块逐项标 ✅/⚠️/❌：
```python
for module in plan_modules:
    status = check_implementation(module)
    # ✅ 已实现 ⚠️ 部分实现 ❌ 完全缺失
```

### Step 3: 三Agent分析
- Agent A(架构): 补齐的技术可行性和依赖
- Agent B(产品): 用户价值排序(P0/P1/P2)
- Agent C(工程): 代码量估算+实施顺序

### Step 4: 分批实施
- Batch1(5项/2周/3350行) → Batch2(5项/2周/5300行) → Batch3(5项/2周/4400行)
- 每批用 delegate_task 的 tasks 数组并行（最多3个）
- 全部Python代码+JS前端+API路由+导航注册一次性写入

### Step 5: 每批验证
- 全部API端点实测（curl真实调用）
- 前端JS文件可访问性
- 边缘情况测试（空数据/坏JSON/超限分页）
- 新增文件必须注册到 app.js PAGE_RENDERERS + index.html 导航 + canvas_web.py 路由

### Step 6: 多轮打磨
- R1: 功能端到端验证
- R2: 边缘情况+压测+错误处理
- R3: 最终全链路验收

## 典型实施量
- 10项功能补齐（Batch1+2）
- ~8,700行新代码
- 9个新引擎模块 + 8个新API路由 + 5个新前端页面
- 方案覆盖度: 40% → ~70%

## 关键教训

1. **使用Python脚本批量修改文件** — 避免 patch replace_all 陷阱
2. **每批完成后立即验证** — 不要攒到最后一起验证
3. **前端JS文件用独立文件** — 不要内联在Python字符串中
4. **路由注册用try/except包裹** — 避免一个子模块失败导致整个服务无法启动
