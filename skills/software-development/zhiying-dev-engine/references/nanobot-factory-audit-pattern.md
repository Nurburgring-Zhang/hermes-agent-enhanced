# Nanobot Factory 极端深度审计模式

## 适用场景
对大型多模块项目（10万+行代码，100+文件，多语言多框架）进行逐行代码审核+全面修复。

## 审计流程（已实战验证145,000行）

### 第1步：全景扫描
```
- 所有文件列表（按目录/行数排序）
- 所有API端点清单（grep @app.get/@app.post等）
- 所有路由模块清单
- 所有核心模块清单
- 所有测试文件清单
- 基础设施文件清单（Docker/.env/alembic等）
```

### 第2步：分批极端深度审核

**每批策略：**
1. 将文件按模块分组（server.py单独一批，core/一批，routes/一批等）
2. 每批3个子Agent并行审核（因delegate_task最大并行3个）
3. 每个子Agent的goal中明确指定文件路径+行数
4. 每个子Agent必须有极端详细的逐行审查要求

**审核维度（13维度）：**
1. API完整性（端点是否全200？返回格式是否一致？）
2. 错误处理（try/except是否到位？全局异常处理器？）
3. 输入验证（Pydantic模型？SQL注入防护？）
4. 认证授权（API Key？RABC？）
5. 配置管理（环境变量？硬编码？）
6. 日志系统（RotatingFile？审计日志？）
7. 性能优化（连接池？缓存？批量操作？）
8. 测试覆盖（可运行？覆盖核心路径？）
9. 部署文档（README？Docker？依赖清单？）
10. UI完整度（前端功能与后端API对齐？Loading/Error/Empty状态？）
11. 数据持久化（SQLite/PostgreSQL？迁移工具？）
12. 安全（CORS？速率限制？HTTPS？）
13. 可观测性（健康检查？metrics？告警？）

### 第3步：真实性验证

**对每个声称"实现了"的功能：**
1. curl实际调用API端点，记录HTTP状态码+返回数据
2. metrics端点验证API调用计数是否与curl次数一致（最强真实性证据）
3. 浏览器实际打开页面点击按钮（验证前端JS是否真实调用后端）
4. 数据库sqlite3检查表和数据行
5. 检查`_generate_simulated_result`/`_mock_*`/`_fake_*`/`random.uniform`等假实现信号

### 第4步：修复优先级

**P0（运行时崩溃/数据丢失/完全不可用）：**
- 表缺失/参数顺序颠倒/索引错误
- async with ClientSession关闭后使用
- asyncio.run()在已有事件循环中调用
- 关键变量未定义
- 路由冲突

**P1（功能错误/接口不匹配/安全风险）：**
- 评分随机数
- 44算子中AI算子不调用模型
- functions壳函数
- agent模块未接入server.py

**P2（代码质量/潜在风险/边缘情况）：**
- 缩进错误
- 注释过时
- 重复代码
- 吞异常

### 第5步：验证标准

- 全项目语法检查 `python3 -c "import py_compile; py_compile.compile(f)"` 对所有修改文件
- pytest全量测试
- 重启服务器后所有端点200
- 浏览器实测关键页面
- metrics计数器与操作一致

## 常见占位符/假实现信号

- `random.uniform` / `random.random` / `random.randint` — 生成假数据
- `_generate_simulated_result` / `_mock_*` / `_fake_*` — 模拟实现壳函数
- `output_file.write_text(f"Generated by...")` — 写入文本冒充图片
- `return f"Executed {func.name}"` — 说执行了但不调API
- `# TODO` / `# FIXME` / `# 暂未实现` — 未完成的占位注释
- `try: ... except: pass` — 吞异常
- `return f"[XX] 完成..."` — 返回描述冒充结果
