# Delivery Checklist

## 何时使用
每次任务完成准备向用户输出最终结果时。

## 清单

### 1. 交付物清单
- [ ] 所有声称创建/修改的文件是否真实存在？
- [ ] 是否有文件路径可以提供给用户验证？
- [ ] 如果是代码交付，构建是否通过？

### 2. 运行证据
- [ ] API是否有实际的curl输出？
- [ ] 测试是否通过（pytest输出）？
- [ ] 前端构建是否通过（npm run build输出）？
- [ ] 是否有HTTP状态码验证？

### 3. 完整度检查
- [ ] 是否有"这个等下一轮"的遗留问题被标记？
- [ ] 是否有已知限制被明确告知？
- [ ] 用户问的所有问题是否都回答了？

### 4. 透明度
- [ ] 是否有隐藏的问题没有被说出来？
- [ ] 是否有"看起来可以但实际上不行"的功能？
- [ ] 如果有降级实现，是否明确标注？

### 5. 下一步建议
- [ ] 给出明确的下一步建议
- [ ] 标记优先级（P0/P1/P2）

## 交付格式
```
## 交付摘要
### 完成项
- xxx (证据: curl输出/构建通过/测试通过)

### 未完成项
- yyy (原因: 需要GPU/需要API Key)

### 下一步建议
- zzz (P0)
```
