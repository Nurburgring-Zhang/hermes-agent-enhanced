# Multi-Vendor System Integration — 多项目深度集成方法论

## 何时使用
当需要将多个第三方开源项目(React/Node.js/Vue/纯前端)的能力深度集成到一个统一Python后端的"超级项目"中时。

## 前置调研流程
1. 下载所有目标项目到 vendor/ 目录
2. 对每个项目执行"项目侦察"(文件数/代码行/技术栈/功能清单)
3. 制作功能对照表,标记重叠和缺口
4. 确定技术栈转换矩阵

## 集成优先级
P0: 核心API层(文件上传/资源库/系统设置/画布管理)
P1: 高频功能(图片处理/云存储/外部Provider/Figma联动)
P2: 扩展功能(主题管理/成就系统/教程模块)

## 物理文件迁移清单(强制步骤)
**源项目不只是代码,还有模板/资源/样式/配置/依赖。** 所有非代码文件必须物理拷贝到目标项目。

### 强制规则: 逻辑复刻不等于完整迁移
这条规则是**多项目集成中最容易被违反的**。违反后果=用户暴怒。

**错误的认知**: 已经用Python重写了API逻辑,所有功能都有了
**正确的认知**: API逻辑重写了,但模板/样式/主题/资源/文件/配置一个都没拷过来,项目只有1MB但源项目有274MB

### 必须物理迁移的内容
源项目文件类型 目标项目路径
模板文件  templates/项目名/
样式文件  frontend/src/styles/(改名加前缀)
静态图片  frontend/public/images/
配置文件  frontend/(改name)
功能清单  data/features/

### 验证命令
迁移前: du -sh PROJECT
vendor: du -sh VENDOR
执行迁移后: du -sh PROJECT(目标大小应接近源项目总和)

### 强制规则: 前端不是附加功能,是核心交付物
后端API写99条路由只是完成了可调用。用户能交互使用的前端界面才是完整交付。

**常见的假装有行为(已被用户暴怒纠正):**
- 写纯静态HTML模板做展示,没有真实交互
- 写输入框但没有提交逻辑
- 声明支持拖拽但没实现拖拽事件
- 有执行按钮但不调用后端API
- 说支持文件上传但前端没有FileReader

**前端验收标准(每一项必须实现):**
1. 节点可从工具箱拖拽到画布
2. 节点在画布上可拖拽移动
3. 节点之间可连线
4. 文件可拖放上传
5. 执行按钮真实调用后端API
6. 工作流可保存为JSON文件
7. 工作流可从JSON文件加载
8. 键盘快捷键删除选中节点
9. 右键菜单操作
10. 执行日志面板实时显示状态

以上任何一项缺失=前端未完成=拒绝交付

## 验证标准(L0-L5)
L0: 所有模块可导入,无ImportError
L1: 文件数/大小接近源项目总和
L2: 模板/样式/资源可查可引用
L3: Web UI启动,HTTP 200,核心API返回数据
L4: 删除vendor目录后项目仍完整独立
L5: 全部原有测试通过(如有)

完成标准: L3至少。只做到L0不算完成。

## 参考案例: IMDF
- 集成来源: Penguin Canvas(155K行/421文件) + html-video + hyperframes + frontend-slides + garden-skills + crawl4ai
- 结果: 36个Python模块/90API路由/679文件/82MB
- 技术栈转换: React19+TS+Electron + Three.js + Express to Python3+FastAPI
- 位置: /mnt/d/Hermes/infinite-multimodal-data-foundry/
