# 用户文档风格指南

**适用场景**: README、项目介绍、commit message、GitHub发布文档

## 核心原则

### 禁止使用的符号
- `##` `**` 标题/加粗标记 — 纯文字叙述，不用markdown标题层级
- `-` 开头列表 — 用段落自然叙述
- `1.` `2.` `3.` 编号 — 不要编号前缀
- `---` 分隔线 — 不要
- `【】` `→` — 不要任何装饰符号
- `#` 标题符号 — 不要

### 语气要求
- 说人话，接地气，直接
- 不要官方腔、不要堆形容词（"全面的""强大的""高效的"）
- 不要寒暄（"欢迎使用""希望你喜欢"）
- 不要AI书面感（"本工具旨在""该节点具备"）
- 短句为主，自然口语

### 高风险操作警告
- 必须大红色大字居中突出
- 加粗、大字号(32px)、红色边框(4px)、红色背景
- `务必先做好备份！` 重复三行

### Commit message
- 极简，一个词或几个字
- 不写描述性长标题
- 如: "README" "修复引用" "精简目录"

### 文档结构
- 先问"这是干啥的"（一句话定位）
- 再写"怎么装"（安装步骤）
- 再写"怎么用"（使用说明）
- 最后许可证
- 不要"A. 引言 B. 背景 C. 概述"这种学院派结构

### 适用文件
- README.md
- 项目介绍页
- GitHub仓库描述
- commit message

## 示例对照

❌ 旧风格（符号多、端着）：
```
# Hermes 全量增强包

**让 Hermes Agent 从"能跑"变成"真他妈能打"**

---

## 这包是干啥的

- ❌ 不会主动调用自己的全量能力
- ❌ 偷懒进行降级实现
```

✅ 新风格（纯文字、说人话）：
```
Hermes Copilot

让 Hermes Agent 从能跑变成真能打

这包是干啥的

Hermes Agent 本身是个通用框架。但默认状态下它：
不会主动调用自己的全量能力
不会主动把复杂任务拆成多段
偷懒进行降级实现
```

删掉所有 - ❌ 和 ** 和 ---。段落自然叙述。
